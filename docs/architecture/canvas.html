Deepening candidates for Codexhub

Module: box with an interface and implementation.

Current read

The core domain language is relatively crisp: projects, workspaces, worker sessions, run groups, review findings, items, and transcripts. The clearest shallow pressure is not in pure core logic. It is in the modules that make every caller know persistence, HTTP, CLI formatting, and worker-observation details at the same time.

Runtime seam

CodexRuntimeController is an earned seam: in-process runtime, supervisor client, and tests already justify multiple adapters.

Core transcript projection

packages/core/src/transcript.ts has a small interface and hides a meaningful implementation. Keep tests at that interface.

Top recommendation

Start with the worker session store

It has the highest leverage because the server, runtime, review gate, transcript surfaces, and tests all touch that seam. Deepening it would improve locality before changing more visible HTTP or CLI interfaces.
