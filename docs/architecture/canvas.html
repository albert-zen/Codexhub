Codexhub - architecture canvas - 2026-05-21

Align Codexhub around the Codex App Server

Module: interface plus implementation.
Seam: where behavior can change by adapter.
Adapter: concrete implementation at a seam.
Deep: small interface, large hidden behavior.

Current read

Codex App Server owns native runtime state: threads, turns, items, and the core agent loop. Codexhub should not duplicate that runtime. Codexhub should own the local control plane: project and workspace context, worker records, run groups, review metadata, raw event mirroring, and operator-facing projections.

System Goal

Codexhub gives a human operator or manager agent a durable, low-context surface for starting Codex workers, observing progress, sending follow-up instructions, grouping runs, and reviewing outcomes without replaying a full Codex thread into context.

Domain Model

Codex-owned state

Thread, turn, item, and core agent runtime. Codexhub stores ids and raw events for correlation, not lifecycle authority.

Codexhub product records

Project, workspace, worker record, task spec, run group, review status, review finding, and message records.

Operator projections

Latest agent message, transcript windows, attention reasons, action availability, and dashboard summaries.

Key Flows

Start Worker

  1. Unified I/O normalizes a CLI or GUI request.
  2. Project Manager validates workspace, task spec, and run group context.
  3. Runtime Ownership Seam starts or connects to Codex App Server.
  4. Adapter starts the native thread and turn.
  5. Raw events append before projections are derived.

Continue Or Steer

  1. Project Manager checks Codexhub state-machine rules.
  2. Runtime Ownership Seam proves the process is live.
  3. Adapter sends the native protocol request.
  4. Events return through Adapter into the raw event log.

Inspect Progress

  1. Planning / Presentation reads records and projections.
  2. CLI and GUI receive bounded, human-readable views.
  3. Raw events remain an explicit debug surface.

Modules And Interfaces

Unified I/O Layer

Interface for CLI and GUI commands plus view/result output. It owns input normalization and output adaptation, so a separate input filter would fail the deletion test.

Project Manager

Owns Codexhub orchestration meaning: worker records, workspace context, run groups, task specs, review state, and follow-up behavior. It must not parse every Codex payload or format UI.

Runtime Ownership Seam

Proves process availability, starts or connects to Codex App Server, and fails closed when ownership is lost. Current CodexRuntimeController is an earned seam.

Codex App Server Adapter

The only Codex-native event ingress. It adapts JSON-RPC requests and appends raw thread, turn, item, approval, and update events as lossless facts.

Codexhub State Substrate

Contains raw Codex event log, Codexhub product records, and derived projections. This is broader and more accurate than WorkerSessionStore.

Planning / Presentation Layer

Converts records and projections into transcript windows, attention reasons, dashboard summaries, available actions, and other operator-facing views.

Risk Map

Project Manager can become a god module if protocol parsing, storage mechanics, and UI formatting move behind one interface.
Adapter can bypass product invariants if it writes anything beyond raw append-only events and narrow normalized outcomes.
Presentation rules can keep splitting across CLI and GUI unless a shared Planning / Presentation module owns operator views.
Codex thread and turn ids must remain foreign references, not Codexhub lifecycle authority.

Decisions

  • Codex App Server owns native thread, turn, item, and agent runtime state.
  • Codexhub owns local orchestration meaning and operator projections.
  • Unified I/O owns input normalization and output adaptation.
  • Raw Codex events belong inside Codexhub State Substrate.
  • State does not feed CLI or GUI directly.
  • Runtime ownership and protocol adaptation are related but separate seams.

Open questions

What to align before implementation

  • Should Planning / Presentation live server-side, or in a shared package consumed by server, CLI, and GUI?
  • Should Project Manager start as one module or a thin orchestrator over worker, workspace, run group, and review modules?
  • Which Codex native events must append before Project Manager updates product records?
  • How should CodexRuntimeController evolve if process supervision and JSON-RPC protocol adaptation are separated?
